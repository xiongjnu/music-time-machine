## 1. 项目概述

### 1.1 一句话定义

一个以时间轴为核心的音乐探索器，用户通过滑动年代 → 选择地区 → 筛选风格（或Mix混合），快速发现和收听每个时代的经典音乐。

### 1.2 核心体验

**滑动 → 发现 → 聆听**，三步完成，零门槛。

### 1.3 项目信息

| 项 | 值 |
|---|---|
| 项目名 | music-time-machine |
| 仓库名 | `github.com/xiongjnu/music-time-machine` |
| 开源协议 | MIT |
| 年代范围 | 1970 - 2020（6个10年跨度） |
| 地区 | 欧美 / 香港 / 大陆 / 台湾 / 日本 / 韩国（6个） |
| 风格 | Mix / 摇滚 / 民谣 / 流行 / R&B / 电子 / 古典 / 嘻哈 / 爵士（9个，含Mix） |

### 1.4 差异化定位

市场上已有项目对比：

| 已有项目 | 做了什么 | 我们的差异 |
|---------|---------|-----------|
| Music Time Machine (GitHub) | Billboard历史榜单+日期选择 | 仅Billboard，无地区/风格筛选，无中国平台 |
| Music Timeline Explorer | D3.js时间线+流派演化可视化 | 没有播放功能，纯可视化 |
| YesPlayMusic (32.8k⭐) | 高颜值网易云第三方播放器 | 没有时间线/年代探索功能 |
| MusicFreeDesktop (8.3k⭐) | 插件化多平台音乐播放器 | 没有时间线，纯播放器 |

**市场空白**：年代探索 × 中国音乐平台 × 地区风格筛选 × 直接播放 = 无人做过

### 2.2 非功能需求

| 编号 | 需求 | 指标 |
|------|------|------|
| NF01 | 启动速度 | 双击脚本后5秒内打开浏览器 |
| NF02 | 歌曲列表加载 | 切换年代/地区/风格后1秒内显示 |
| NF03 | 播放启动延迟 | 点击播放后2秒内开始播放 |
| NF04 | 内存占用 | 后端服务<200MB |
| NF05 | 浏览器兼容 | Chrome 90+, Safari 15+, Firefox 90+ |
| NF06 | 无离线模式 | 必须联网+登录 |
| NF07 | 错误日志 | 播放失败/API错误有分级日志，前端错误自动上报后端 |
| NF08 | 数据冷启动 | 首次启动加载状态<3秒显示，超时10秒降级为有限功能 |
| NF09 | Safari兼容 | 播放需用户手势触发，自动播放限制优雅降级 |

### 2.3 不做的功能

- 用户收藏/歌单功能
- 播放历史记录
- 账号密码登录（仅二维码）
- 离线模式
- 社交分享
- QQ音乐音源接入（已移除，网易云曲库已覆盖）
- 年代二级年份选择器（已移除，年代选择简化为10年节点）
- Electron桌面端（后续版本考虑）
- AI推荐功能（后续版本考虑）

### 3.4 Mix模式 vs 指定风格模式

| 行为 | Mix模式（默认） | 指定风格模式 |
|------|----------------|-------------|
| **选中状态** | 默认选中 | 用户手动点击 |
| **歌曲来源** | 该年代×地区下所有风格，每类最多抽4首，加权随机共30首 | 该年代×地区×风格，加权随机30首 |
| **排序** | 随机打乱（Fisher-Yates） | 随机打乱 |
| **风格标签** | 每首歌显示风格小标签（🎸🎻🎤🎶🎹等） | 风格统一，不显示标签 |
| **切换行为** | 切回Mix时重新随机组合 | 每次切换重新随机 |

### 3.6 交互细节规则

| 操作 | 效果 | 动画 |
|------|------|------|
| 拖动时间滑块 | 实时切换年代，歌曲列表刷新 | 光带加速流动，列表淡入淡出 |
| 切换地区 | 歌曲列表按新地区刷新 | 列表向左滑出，新列表从右滑入 |

| VIP歌曲点击 | Toast提示"VIP歌曲，正在切换音源" | Toast 2秒后消失 |
| 无音源歌曲 | 显示"暂无音源"，1.5秒后自动跳下一首 | Toast提示 |
| 点击退出 | 清除内存Cookie，回到登录页 | 页面过渡 |

### 5.3 启动时验证机制

```
启动应用
  │
  ▼
读取 data/songs.json
  │
  ▼
遍历所有 active 歌曲 ID
  │
  ▼
批量调用 /song/detail 验证（并发控制：5个/批）
  │
  ├─ <5秒完成 → 全量可用，无额外提示
  ├─ 5-10秒 → 先返回已验证的歌曲，后台继续验证剩余
  ├─ >10秒超时 → 不等待验证结果，直接展示原始JSON数据（未验证状态）
  │    前端显示"部分歌曲状态待确认" Toast
  │
  │   对每首歌的验证结果:
  ├─ 返回有效 → 保留
  ├─ 返回404/无数据 → 标记失效
  │     │
  │     ▼
  │   从 reserve 中取候补顶入 active
  │     │
  │     ▼
  │   更新内存缓存（不修改JSON文件）
  │
  └─ reserve 也耗尽 → 该格子标记 dataInsufficient: true
                        前端显示"该风格暂无足够歌曲"
```

**关键设计**：

| 设计项 | 方案 | 理由 |
|--------|------|------|
| 验证频率 | 仅启动时一次 | 避免频繁请求触发风控 |
| 结果存储 | 写入内存缓存，不修改JSON | 避免文件写入磨损；JSON是策展源文件，不应被程序修改 |
| 并发控制 | 5个/批，批间间隔500ms | 控制请求频率，不触发限速 |
| 超时处理 | <5秒全量可用 / 5-10秒分批返回 / >10秒直接展示原始JSON | 防止验证过程阻塞用户操作 |
| 失效日志 | 记录到日志文件 | 便于后续人工补充数据 |
| 降级策略 | 数据不足时坦诚提示用户 | 不回退到实时搜索，坚持策展纯度 |

#### 5.3.1 缓存清理策略

| 数据 | 位置 | 清理策略 |
|------|------|---------|
| 策展数据JSON | `data/songs.json` | 手动管理（项目文件，不被程序修改） |
| 验证结果缓存 | 内存 | 每次重启时重建，不清除文件 |

| 工具脚本临时文件 | 无 | 不产生临时文件 |

### 5.4 加权随机推荐算法

```javascript
function _weightedRandomSelect(pool, count, { maxPerArtist = 2, minYearSpread = 3 } = {}) {
  // 1. 构建抽奖池：每首歌按 weight 值出现对应次数
  const lottery = [];
  for (const song of pool) {
    const tickets = song.weight || 5;
    for (let i = 0; i < tickets; i++) lottery.push(song);
  }

  // 2. 贪心随机选取，满足约束
  const selected = [];
  const artistCount = new Map();
  const years = new Set();

  while (selected.length < count && lottery.length > 0) {
    const idx = Math.floor(Math.random() * lottery.length);
    const song = lottery[idx];

    const artistCnt = artistCount.get(song.artist) || 0;
    if (artistCnt >= maxPerArtist) {
      // 移除该歌手所有抽奖票
      lottery.splice(idx, 1);
      continue;
    }

    selected.push(song);
    artistCount.set(song.artist, (artistCount.get(song.artist) || 0) + 1);
    if (song.year) years.add(song.year);

    // 移除该歌曲所有抽奖票
    for (let i = lottery.length - 1; i >= 0; i--) {
      if (lottery[i].id === song.id) lottery.splice(i, 1);
    }
  }

  // 3. 后处理 swap 优化年份分散度
  // ...

  // 4. Fisher-Yates 打乱
  return _shuffle(selected);
}

function getSongs(era, region, genre) {
  if (genre === 'mix') return _generateMix(era, region);
  const slot = slots[`${era}-${region}-${genre}`];
  return _weightedRandomSelect(slot.active, 30);
}

function _generateMix(era, region) {
  // 每 genre 最多取4首，加权随机共30首
  const allSongs = [];
  for (const genre of GENRES) {
    const slot = slots[`${era}-${region}-${genre}`];
    if (slot?.active?.length) {
      const picked = _weightedRandomSelect(slot.active, 4);
      allSongs.push(...picked);
    }
  }
  return _weightedRandomSelect(allSongs, 30);
}
```

- Firefox 90+：Web Audio API限制少于Safari，无特殊处理

- 所有浏览器：`<audio>` 标签使用 `preload="none"`，由用户点击触发加载

## 7. 安全与风控

### 7.1 网易云风控风险评估

> **核心结论：风控风险存在但可控。正常听歌（每天登录2-3次，手动操作，非自动化）属于中低风险，不是"用了就封"。真正被严打的是刷量/自动化脚本，不是普通播放行为。**

#### 7.1.1 风控检测机制

网易采用**多层检测系统**，不是单一因素触发：

| 检测层 | 检测手段 | 技术细节 |
|--------|---------|----------|
| **第1层：登录身份验证** | 客户端指纹检测 | 识别请求是否来自官方客户端，拦截非官方请求 |
| | 易盾(YiDun)验证 | 网易反欺诈SDK集成在登录流程，第三方客户端不支持此验证步骤 |
| | EApi加密头检测 | 网易加密API(eapi)有专用Header集，第三方客户端通常缺失 |
| | 错误代码8821 | 返回"需要行为验证码验证"，检测到非官方登录时触发 |
| **第2层：请求模式分析** | 请求频率分析 | 检测非人类播放速度（刷歌检测） |
| | IP信誉评分 | 云服务器IP被特别标记；未知/不常见IP触发风控 |
| | Cookie使用模式 | 检测来自第三方工具的Cookie认证模式 |
| **第3层：行为分析** | 播放模式检测 | "非法挂机行为"——检测无人值守的自动播放 |
| | 签到异常检测 | 多IP签到（签到脚本使用不同服务器）触发检测 |
| | 设备变化检测 | 频繁更换登录设备/IP触发风控 |

#### 7.1.2 不同行为的风险等级

| 行为场景 | 风险等级 | 典型后果 | 证据来源 |
|---------|---------|---------|----------|
| 批量刷歌（300首/天自动播放） | 🔴 极高 | "非法挂机行为"警告→封号 | Go SDK明确警告 "风控极为严格" |
| 第三方客户端+用户名密码登录 | 🔴 高 | 强制下线→改密码→3天冻结 | SPlayer、Melody项目确认 |
| 第三方客户端+二维码扫码登录 | 🟡 中 | 部分检测，比密码登录风险低 | YesPlayMusic用户报告 |
| 第三方客户端+Cookie/CookieCloud登录 | 🟢 中低 | 目前最安全的登录方式 | Go SDK推荐方案 |
| 正常听歌（登录后低频手动播放） | 🟢 低 | 登录后播放行为检测难度大 | 多源交叉验证 |
| 下载歌曲 | 🟢 低 | 不被视为风险行为 | Go SDK未标记任何风险 |
| 搜索歌曲 | 🟢 低 | 不需登录即可使用 | - |
| 获取榜单 | 🟢 低 | 不需登录即可使用 | - |

#### 7.1.3 封号是递进式的

网易不会一步到位永久封号，而是递进式惩罚：

```
首次触发 → 强制下线 + 要求改密码
     ↓ 再次触发（继续使用第三方客户端）
3天冻结
     ↓ 继续触发
7天冻结
     ↓ 屡次不改
更长时间冻结
     ↓ 极端情况（极罕见）
永久封号
```

> **永久封号极罕见**，多数用户在3-7天冻结后可恢复。

#### 7.1.4 真实封号案例

| 时间 | 平台 | 情况 | 后果 |
|------|------|------|------|
| 2024年10月 | YesPlayMusic用户 | 扫码登录后正常听歌 | 账号3天冻结 |
| 2024年12月 | YesPlayMusic用户 | 10月被封后继续用 | 再次冻结，用户表示"搞怕了" |
| 2026年4月 | 大规模事件 | 大量用户（含官方客户端用户）被冻 | 网易称"系统异常"，后解封 |
| 酷安用户 | 第三方客户端登录 | 收到"涉嫌违规"冻结通知 | - |

> **注意**：2026年4月大规模封号事件中，连官方客户端用户都中招——说明风控系统有时会误伤，不只有第三方用户会被冻。

#### 7.1.5 我们项目的风险定位

本项目典型用户场景：**主号登录，每天2-3次，正常听歌**。

| 环节 | 风险 | 说明 |
|------|------|------|
| **登录环节** | 🟡 中 | 二维码扫码比密码登录安全，但仍是主要检测点 |
| **搜索环节** | 🟢 低 | 无需登录，不触发风控 |
| **播放环节** | 🟢 低 | 正常频率播放，检测概率极低 |
| **数据验证** | 🟢 极低 | 仅启动时批量验证歌曲ID，请求量极小 |

**总结**：登录是唯一的中风险环节。一旦登录成功且未被即时拦截，后续正常播放几乎不会被检测。

### 7.2 三层防护体系

#### 第1层：降低登录检测风险（最关键）

| 措施 | 实现方式 | 优先级 |
|------|---------|--------|
| 二维码扫码登录 | 已确认方案，比密码登录安全 | P0 |
| 登录后保持会话 | Cookie存内存，不频繁重新登录，会话过期前持续使用 | P0 |
| 模拟官方客户端Header | 参考HyPlayer做法，在关键请求中伪造User-Agent等Header | P0 |
| 登录频率限制 | 同一账号5分钟内最多登录1次 | P1 |

#### 第2层：降低播放行为检测风险

| 措施 | 实现方式 | 优先级 |
|------|---------|--------|
| 请求限速 | 控制API调用频率，不超过正常人类节奏（60请求/分钟） | P0 |
| 数据缓存 | 搜索结果缓存24小时，播放URL缓存2小时，减少重复请求 | P0 |
| 不自动播放 | 进入页面不自动播放，需用户手动点击 | P0 |
| 无刷歌功能 | 不做"自动播放全部"之类的自动化功能 | P0 |

#### 第3层：高级防护（后续迭代）

| 措施 | 实现方式 | 优先级 |
|------|---------|--------|
| Header注入（HyPlayer方案） | 用户通过mitmproxy从官方客户端捕获真实Header/EApiHeaders，注入到后端 | P2 |
| CookieCloud同步 | 支持从CookieCloud导入官方客户端Cookie，避免直接登录 | P2 |
| 只读降级模式 | 如果登录风控过严，可降级为无登录模式（仅搜索+榜单，不可播VIP歌曲） | P2 |

### 7.3 登录安全设计

| 设计项 | 方案 | 理由 |
|--------|------|------|
| 登录方式 | 仅二维码扫码 | 比密码登录更安全，风控检测率更低 |
| Cookie存储 | 仅内存，不落盘 | 关闭浏览器即失效，减少泄露风险 |
| Cookie有效期 | 会话持续期间保持有效 | 减少重复登录次数 |
| 无自动登录 | 每次启动需重新扫码 | 避免长期存留Cookie |
| 风险提示 | 登录页温和提示 | "本项目为第三方客户端，极小概率触发平台风控" |

### 7.4 前端安全

| 设计项 | 方案 |
|--------|------|
| XSS防护 | 所有动态内容转义 |
| CSRF | SameSite Cookie |
| API密钥 | 无需API Key，所有请求走后端代理 |

### 7.5 数据冷启动设计

| 场景 | 前端行为 | 后端行为 |
|------|---------|---------|
| 首次启动（验证中） | 显示"时光隧道加载中..."动画 | 读取JSON + 批量验证 |
| 验证超时（>10秒） | Toast提示"部分歌曲状态待确认"，显示已加载数据 | 停止验证，返回当前缓存 |
| 验证失败 | Toast提示"数据加载异常"，提供重试按钮 | 返回原始JSON数据（未验证） |
| 数据为空（某格子无歌曲） | 显示"该年代暂无歌曲"引导用户切换 | 返回空数组 |
| 验证全部完成 | 无额外提示，正常使用 | 返回已验证数据 |

### 12.5 本地生产部署

**部署方案**: PM2 进程管理 + macOS launchd 开机自启

```bash
# 启动
pm2 start ecosystem.config.js

# 状态查看
pm2 status
pm2 logs musictm

# 重启/停止
pm2 restart musictm
pm2 stop musictm

# 开机自启配置（需sudo）
sudo env PATH=$PATH:/usr/local/bin $(which pm2) startup launchd -u xj --hp $HOME
pm2 save
```

```js
module.exports = {
  apps: [{
    name: 'musictm',
    script: 'server/index.js',
    cwd: '/Users/xj/projectxjai/claude/code/musictm',
    env: { PORT: 3000, NODE_ENV: 'production' },
    max_memory_restart: '200M',
    autorestart: true,
    watch: false,

};
```
